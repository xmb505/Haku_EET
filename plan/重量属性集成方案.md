# 电梯重量属性集成方案

> 状态：规划完成，待实施
> 最近更新：2026-07-15

## 目标

为每部电梯的 `Car` 实体引入**三态重量属性**（0=正常 / 1=临界 / 2=超重），由**小脑**维护状态机，**大脑**只读状态决策；配套引入：

1. **三 config 架构**（`config/io_profile/` 目录化 + 冷切换）
2. **IO2HTTP 独立 `/word_read` 端点**（与 `/gpio` 隔离数字量/模拟量）
3. **三态机 + 应急 polling**（状态 2 期间每 200ms 查 word，weight ≤ max 退出）
4. **复用 `cancel_for_reopen` 流程**（状态 2 紧急开门与开门按钮按下同流程）
5. **满载灯（`full_load_indicator`）与外召 LED 完全解耦**

---

## 一、IO2HTTP 端（用户先做）

### 1.1 端点隔离动机

`/gpio` 设计之初只考虑数字量（bit），未考虑模拟量（word）。两者时序/语义不同：
- 数字量：PLC bitmap 主动推送（WS），写命令是离散 seter
- 模拟量：需要程序主动问（HTTP reader），单值读取，无需 bitmap

**必须隔离**，否则 IO2HTTP 内部 S7 读路径、缓存、日志都会乱。

### 1.2 新增 `/word_read` 端点

```http
POST http://127.0.0.1:8080/word_read
Content-Type: application/json
{"alias": "weight", "db_num": 10, "byte": 28, "count": 1}

→ 200 OK
{"success": true, "alias": "weight", "db_number": 10,
 "readings": [{"byte": 28, "word": 950, "hex": "0x03B6"}]}
```

**参数说明**：
- `alias`: 控制器别名（config.ini `[Weight_Read]` 段，默认 `weight`）
- `db_num`: DB 块号（可覆盖 config 默认值）
- `byte`: 起始字节地址（可覆盖 config 默认值）
- `count`: 读取 WORD 数量（默认 1）

**响应**：
- 成功：`success=true`, `readings` 为 `[{byte, word, hex}, ...]` 数组
- 失败：`success=false`, `error` 字段说明原因
- readings 为空数组也视为失败

**实现要点**：
- 独立 S7 读路径（不与 `/gpio` 共享缓存）
- 可选：50ms 缓存（同 PLC 扫描周期内复用结果）
- 失败响应：`{"value": -1, "error": "..."}` 或直接 HTTP 4xx

## 三、重量三态机设计

### 3.1 三态语义表

| state | 触发条件 | 大脑(passenger)行为 | 小脑副作用 |
|-------|----------|----------------------|------------|
| **0 正常** | `weight < max*threshold` | 正常响应外呼/内呼/顺路 | 无 |
| **1 临界** | `max*threshold ≤ weight ≤ max` | **不响应外呼**,内呼+顺路照常 | 满载灯**不亮** |
| **2 超重** | `weight > max` | 完全不可用(不是派车候选) | 复用 cancel_for_reopen 开门 + 满载灯亮 + 启 polling |

### 3.2 状态机转移 + 副作用

```
                 weight ≥ max*threshold
        ┌────────────────────────────────┐
        │  0 正常  ────────────────────▶  1 临界
        │   ▲                             │
        │   │                             │  weight > max
        │   │                             ▼
        │   │                          2 超重
        │   │                          ├─ 复用 cancel_for_reopen 开门
        │   │                          ├─ 满载灯亮(full_load_indicator=true)
        │   │                          └─ 启动 polling(每 200ms 查 word)
        │   │                             │
        │   │  polling 读到 weight ≤ max  │
        │   │  (可能降为 1 或 0)          │
        │   └─────────────────────────────┘
        │       ├─ weight > threshold → state=1, 熄满载灯, polling 结束
        │       └─ weight ≤ threshold → state=0, 熄满载灯, polling 结束
```

### 3.3 三个状态更新钩子(小脑触发 read_word)

| 钩子 | 触发时机 | 副作用 |
|------|----------|--------|
| **H1** 关门动作开始 | executor 收到 CLOSE_DOOR action | 查 word → 更新 state;**state=2 时复用 cancel_for_reopen 开门 + 亮满载灯 + 启 polling** |
| **H2** 关门动作完成 | executor 完成 CLOSE_DOOR | 查 word → 更新 state(0↔1 切换) |
| **H3** 开门按钮按下 | door_open_button bit=1 | 查 word → 更新 state(仅同步) |

### 3.4 复用 cancel_for_reopen 流程(状态 2 紧急开门)

```python
# 在 H1 钩子内,状态 2 触发时
async def _handle_weight_state_2(self, cid: int):
    car = self.cars[cid]
    if car.door_state in (DoorState.CLOSING, DoorState.CLOSED):
        # 复用现有"开门按钮打断"组合(passenger.on_door_button 用同一段)
        self.executors[cid].door.cancel_for_reopen()
        await self.action_queues[cid].put(Action(ActionKind.OPEN_DOOR))
    # 满载灯亮
    await self.ui[cid].set_full_load(True)
    # 启动应急 polling
    self._start_weight_emergency_poll(cid)
```

> **「复用」的边界**：状态 2 的入口逻辑（cancel_for_reopen + OPEN_DOOR 入队）跟现有开门按钮按下场景完全一致——但调用方从 passenger 改为 weight_manager。passenger 的现有 reopen 逻辑完全不动。

### 3.5 应急 polling 实现

```python
# core/weight_manager.py
async def _weight_emergency_poll(self, cid: int):
    """仅在 state=2 期间运行,polling 读到 weight ≤ max 即结束"""
    car = self.cars[cid]
    interval = self._poll_interval_s()
    while True:
        await asyncio.sleep(interval)
        weight = await self._app._read_car_weight(cid)
        if weight == -1:  # read_word 失败,继续等
            continue
        if weight <= car.max_weight:
            threshold = int(car.max_weight * car.weight_threshold)
            car.weight_state = 1 if weight > threshold else 0
            await self._app.ui[cid].set_full_load(False)
            return  # 任务自然结束
```

> 事件驱动哲学的工程例外(与 `executor.py:443-446` brake-before-stop 100ms sleep 同类):polling 只在 state=2 期间运行,有明确退出条件。

### 3.6 防止「开门 → 关门 cron → 立即再开」循环

```python
# core/passenger.py _close_door_action 内
async def _close_door_action():
    car = self._app.cars[car_id]
    if car.door_state not in (DoorState.OPEN, DoorState.OPENING):
        return
    # ★ 新增:状态 2 期间不调度关门
    if car.weight_state == 2:
        return
    # ... 现有逻辑(光幕/外召按钮检查)...
```

### 3.7 满载灯 vs 外召 LED(完全解耦)

| 事件 | full_load_indicator | 外召 LED(如 L5 上) |
|------|---------------------|----------------------|
| 状态 0 | 灭 | 现有逻辑(按下亮+松开看门状态) |
| 状态 1 | 灭 | 现有逻辑(按下亮+松开看门状态) |
| 状态 2 触发瞬间 | **亮** | 现有逻辑(按下亮+松开看门状态) |
| 状态 2 polling 降级 | 灭 | 现有逻辑 |
| 状态 2 期间外召按下 | **亮**(满载灯独立) | 亮(外召) |
| 状态 2 期间外召松开 | **亮** | 按门状态(可能仍亮) |

> **解耦确认**:满载灯是电梯自身状态的反映;外召 LED 是用户交互的反映。两条独立信号。

## 五、整体调用链

```
PLC weight Word 变化(只在以下时机被读取):
  │
  ├─ 关门动作开始(executor 收到 CLOSE_DOOR)
  │   └─ read_word(cid) → 更新 car.weight_kg
  │       └─ 计算新 weight_state
  │           ├─ 0/1: 正常关门
  │           └─ 2: 拦截 → cancel_for_reopen + OPEN_DOOR + 亮满载灯 + 启 polling
  │
  ├─ 关门动作完成(executor 完成 CLOSE_DOOR)
  │   └─ read_word(cid) → 更新 weight_state
  │       └─ 仅 0↔1 状态切换可能
  │
  ├─ 开门按钮按下(door_open_button bit=1)
  │   └─ read_word(cid) → 更新 weight_state
  │       └─ 仅作状态跟踪,不影响开门动作
  │
  └─ [应急] 状态 2 polling 期间
      └─ 每 200ms read_word → 检查降级 → 退出 polling
```

## 六、关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 端点隔离 | `/word_read` 与 `/gpio` 分开 | 数字量/模拟量时序语义不同,IO2HTTP 内部 S7 路径独立 |
| profile 切换模式 | **纯冷切换**(启动时选,改完需重启) | 避免半切换半不一致状态 |
| 状态机维护者 | **小脑**(executor + ui 协作) | 游戏化编程——重量是电梯属性,小脑管属性 |
| 状态机决策者 | **大脑**(passenger) | 只读 car.weight_state,不做 read_word |
| 状态数 | **三态** 0/1/2 | 0 正常 / 1 临界不接外呼 / 2 超重立即开门 |
| 状态 2 开门 | **复用 cancel_for_reopen** | 与开门按钮按下流程同,避免另写一套 |
| polling 退出阈值 | **weight ≤ max** | 退出时 state 可能为 1(weight > threshold)或 0(weight ≤ threshold) |
| 满载灯与外召 LED | **完全解耦** | 满载灯反映电梯自身状态,外召 LED 反映用户交互 |
| simulate 模式 | **VPLC.read_word 返 0** | 用户不用 simulate,简化实现,state 永远 0 |
| /weight REPL 命令 | **不实现** | 用户在 TIA 里手动设置 DBW28 |
| 写 word 协议 | `POST /word_read` 同步请求 | 不走 WS bitmap,程序主动问 |
| polling 频率 | **200ms** (5Hz) | 状态 2 期间才运行,exit condition 明确 |

| 场景 | TIA 设值 | 预期 state | 预期副作用 |

### 7.0 /car status 重量显示

`/car <id> status` 命令执行时：
- 触发 `app._read_car_weight(cid)` 一次（向 IO2HTTP 主动查询最新 word）
- 输出增加 `重量: <kg> kg (state=<0|1|2> <标签>)`
- state 标签：`0=正常` / `1=临界` / `2=超重`
- read_word 失败时显示 `重量: N/A (read 失败)`

```
haku> /car 1 status

门状态:      closed

显示:        5

```

**实现位置**:`core/console.py:655 _do_status` 方法内,`snap = self.app.status_snapshot(...)` 之前插入 `await self.app._read_car_weight(requested)`,然后在 print 块中加一行。`app.status_snapshot` 返回的 car dict 需包含 `weight_kg` / `weight_state` 字段。

### 7.2 状态 2 循环拦截

- 状态 2 期间关门 cron 到点 → 不调度关门(避免循环)
- 状态 2 期间再按关门按钮 → 不响应

### 7.3 外召 LED 独立

- 状态 2 期间外召 LED 行为完全不变
- 满载灯独立于外召 LED

### 7.4 fail-safe

- read_word 超时 → 视为超载(state=2)
- HTTP 不可达 → state=2
- profile 无 weight_word 配置 → 跳过 read_word,state 永远 0

### 7.5 profile 隔离

- local_train profile: `addr_word_input('weight', 1)` → KeyError
- competition profile: `addr_word_input('weight', 1)` → 'DB10.DBW28'
- local_with_weight profile: `addr_word_input('weight', 1)` → 'DB10.DBW28'

### 7.6 profile 冷切换

- 启动时 `io_profile=local_train` → 加载 local_train.yaml
- 改成 `competition` 后 `/reload` 不切换,打 warning
- 重启后生效,启动日志显示 profile 名

### 7.7 simulate 模式

- VPLC.read_word 返 0 → state 永远 0
- 满载守卫不触发(预期行为,符合"不用 simulate"前提)
