本仓库采用分层防御式错误处理策略，围绕 PLC 硬件控制场景设计，核心思想是异常即安全——任何不可恢复的错误都导向 FAULT 状态并触发紧急停止。

## 1. 错误类型与传播机制

- 自定义数据类故障标志: player.py 中的 FaultFlags 数据类集中记录过载、检修、光幕、上下限位等传感器状态，通过 dataclasses.replace() 原子更新，避免竞态。
- 状态机驱动: CarState（UNKNOWN/READY/FAULT）和 DoorState 作为全局错误可见性层，所有高层 API 在操作前检查状态，禁止对 FAULT 车执行非手动命令。
- 返回值编码错误: 高层 API 不抛异常而返回结构化结果：如 change_internal 返回 'accepted'/'rejected'/'not_running'，fireman 返回包含 status 的 dict，便于 console 直接展示。
- ValueError 用于参数校验: set_hall_indicator 等方法对非法楼层/方向直接 raise ValueError，由 console 统一捕获打印。

## 2. 关键错误路径

### 紧急停止（最严重错误）
executor._emergency_stop() -> 清所有接触器 + 置 FAULT + 取消门动作 + 清理站点吸附状态
触发条件：
- 2 限位（坠机限位）被触发：on_io_event 中优先于所有逻辑检查
- 用户 /car N stop 或 manual_emergency_stop
- 任何丢分危险情况

### IO 层容错
- IOClient._flush_now: HTTP POST 失败时 raise IOError，上层任务会因异常退出
- _ws_loop: WebSocket 断开后自动重连（reconnect_delay=3s），期间 ws_connected=False
- _dispatch: 监听器回调异常仅 print 日志，不影响其他监听器
- simulate_input: 非 simulate 模式调用直接 raise RuntimeError

### 配置与映射错误
- IOMapper.addr_output/input: 信号名不存在时 raise KeyError，调用方用 try/except 优雅降级
- app.set_usermode: 未就绪轿厢 ID 放入 blocked 列表而非抛异常

## 3. 异步错误隔离

- listener 隔离: IOClient._dispatch 逐个 await 监听器，单个异常不影响后续
- 任务取消安全: stop() 中 cancel 所有后台 task 并 await 捕获 CancelledError
- Future 超时保护: _level_seek_check 使用 asyncio.wait_for(..., timeout=3.0) 防止反冲死锁
- finally 清理: 手动模式退出时恢复 executor.paused 状态，确保下次自动调度正常

## 4. 开发者约定

1. 硬件相关操作必须 try/except KeyError：所有 mapper 地址查找都可能因配置缺失失败
2. FAULT 状态不可逆：只有 manual 模式推出 2 限位后才能自动恢复 READY
3. 不要依赖异常流控制业务逻辑：业务分支用返回值编码（如 'accepted'/'rejected'）
4. 紧急停止优先级最高：任何 IO 事件路径都要先检查 2 限位
5. 异步边界处显式捕获异常：listener、cron job、后台任务中的异常只记录不传播
