模块按作者定义的「大脑-小脑-脑干」三层单向依赖组织：
- 大脑层 `passenger.py`：纯流程管理（关门 cron、乘客队列 PassengerQueue、外召灯闪烁），不注册任何 IO 监听器，仅通过 app.py 暴露的 API 与小脑交互；拥有独立于小脑 pending_calls 的乘客请求队列。
- 小脑层 `app.py` + `executor.py` + `ui.py` + `controllers.py`：app 装配每部 Car/Executor/ActionQueue/VirtualPLC，共享 IOClient/IOMapper/Algorithm；executor 是硬件 FSM（IDLE/MOVE/INITIALIZE/DOOR/...），从 ActionQueue 取动作并展开为 IO 序列，监听 level_up/down、限位等传感器推进状态；ui 是纯 IO 写映射器，只从 Car.ui 读状态并通过 set_many 写入 IO，不监听事件；controllers 封装 MotorController/DoorController，直接操作 PLC 接触器/继电器/刹车信号。
- 脑干层 `io_client.py` / `io_mapper.py` / `display.py` / `player.py` / `virtual_plc.py`：IO 抽象、地址映射、7 段显示编码、Car 实体模型、虚拟 PLC 模拟。
- 依赖方向严格单向：大脑 → 小脑 → 脑干，不允许反向；cron 作为外置耳机通过 EventRule 做 reschedule/cancel，不属于任何层。
- 多轿厢设计：App 为每部车维护独立的 io_write(IOClient) 实例，共享 self.io 的 input/output cache，避免 S7 read-modify-write 时一次 POST 30+ 地址导致的接触器建立时间冲突。
- slow_brake 功能：在 controllers.MotorController.set_speed(high_speed=False) 时根据 slow_brake_level 位掩码叠加 brake_1/2/3，由 app 启动时从 config['elevator']['slow_brake'] 注入到每个 executor.motor。
