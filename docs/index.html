§1

三大核心设计哲学elevator-as-player · three-layer separation · event-driven

Haku_EET 不是又一套"读 PLC、写 PLC、循环"的电梯程序——它把硬件细节彻底封进底层， 让代码读起来像在玩 Minecraft 的服务端插件。这种抽象不是技术炫技，而是为了： 在比赛现场快速调整、快速验证、快速交接。

游戏化编程

电梯是玩家（Car），有属性（楼层/方向/门/故障/UI）， 有数值，有状态。大脑只修改属性，小脑自动把属性同步到物理世界。

"改 position，引擎自动渲染——这是 Haku_EET 的设计基线。"

三层严格分离

大脑决策 / 小脑物理执行（运动+用户交互） / 脑干IO 抽象。不允许跳层、不允许跨调用、不允许参数泄漏。

"小脑不知道调度，大脑不知道 IO 地址。"

事件驱动

没有 time.sleep、没有 while: poll。每一层广播自己的消息、监听别人的消息。 cron 是不属于任何层的"外置耳机"——以"定闹钟"取代"数秒"。

"事件驱动不是技术选择，是架构承诺。"

§3

8 条设计不变量non-negotiable invariants

写代码、写文档、Code Review 时这 8 条是红线——任何时候不允许违反。 违反任何一条都意味着"这不属于 Haku_EET 的代码风格"。

01 三层分离

大脑 / 小脑 / 脑干严格分离。不允许跳层、不允许跨调用、不允许参数泄漏。

✕ 反例：算法层 import io_mapper · 小脑直接写 DB 地址

02 电梯 = 游戏实体

Car 是玩家，有属性有数值有状态。绝不包含 IO 地址。

✕ 反例：往 Car 里加 io_address / plc_tag 字段

03 游戏属性驱动

大脑修改 Car 属性，小脑自动同步到物理世界。大脑不碰 IO，小脑不碰调度。

✕ 反例：算法层直接调 io.set(...)

04 算法层是调度员

输入全局状态 + 全局需求，输出"谁去哪 / 谁变更目的地"。不做运动学，不做 IO 读写。

✕ 反例：decide() 返回接触器拉合序列

05 事件驱动

整个系统没有 asyncio.sleep(N) 做"等 N 秒后干什么"，没有 while: check()。

✕ 唯一例外：executor 刹车前 100ms（PLC 物理时序必备）

06 cron 是外置耳机

cron 不属于任何层，挂在事件总线上。EventRule 两种模式：reschedule / cancel。

✕ 反例：把 cron 写进小脑或大脑内部

07 不外设逻辑不污染调度

用户交互模块只翻译需求，算法层只做调度。两者不跨界。

✕ 反例：算法层直接读 IO 判断按钮

08 点位表 = IO 真相来源

tools/gen_io_config.py 解析 点位表.md 生成 io_config.yaml。代码不硬编码地址。

✕ 反例：代码里写 DB11.DBX6.0

§4

cron——外置耳机events, not sleeps

"等 10 秒后关门"这件事，在 Haku_EET 里不是await asyncio.sleep(10)， 而是定一个 10 秒的闹钟——它能被光幕挡住、被人按开门按钮打断、被新的需求重置。

⏰ 核心数据结构

core/cron.py 
CronJob（任务）
  ├─ trigger_time    闹钟响的绝对时间
  ├─ action          闹钟响时执行的回调
  ├─ auto_remove     触发后自动清除
  └─ event_rules     哪些事件影响这个闹钟

EventRule（事件规则）
  ├─ signal_name     IO 信号名（'light_curtain' / 'cabin_button_3'）
  ├─ car_id          哪部车
  ├─ action          'reschedule' | 'cancel'
  └─ delay           仅 reschedule：新延时

run 循环——零轮询

while running:
  fire 所有已过期的 job → 自毁 auto_remove
  wait_for(wakeup_event, timeout=next_deadline)

# 只在三种情况下唤醒：
# 1. 定时到期（wait_for timeout）     → fire job
# 2. 新任务 schedule（wakeup_event.set）→ 可能比当前等待更早
# 3. IO 事件触发规则（_on_io_event）   → 重调度/取消

关门场景完整序列

大脑用户交互模块收到"开门完成"事件 → 调度关门闹钟（10s）
  ├─ 有人按开门按钮 → cancel 闹钟（别关了）
  ├─ 按钮松开 → 调度新的关门闹钟（重新数 10s）
  ├─ 光幕被挡（light_curtain=1）→ reschedule（再推 10s）
  └─ 闹钟静悄悄走到响 → 大脑监听到 → 翻译为"需求：关门"
      → 算法层 → Action(CLOSE_DOOR) → 小脑 executor
§5

端到端调用链——以"内召 5 楼"为例event-driven end-to-end

一个按钮按下后，事件如何穿越 5 个处理层、最终驱动 6 个继电器——这就是 Haku_EET 的全部秘密。

1
脑干

IO 客户端收 WS bitmap

PLC 上 cabin_button_5 = 1 变化通过 WebSocket 推上来。
io_client.py 解析 bitmap，按信号名查 io_mapper，广播 (1, 'cabin_button_5', True)。

2
小脑 UI → 大脑用户交互

翻译为需求事件

小脑 UI 模块更新 Car.pending_calls。
大脑用户交互模块（app.py）收到 IO 事件 → 翻译为"1 号梯需要去 5 楼" → 广播需求事件。

3
大脑 算法层

调度 → Action(MOVE_UP)

事件唤醒 _tick()，调用 algorithm.decide(car, pending_calls)。
返回 [Action(MOVE_UP)]。action_queues[1].put(MOVE_UP)。

4
小脑 executor

Action → IO 序列

从 ActionQueue 取到 MOVE_UP。
拉上行接触器 + 高速接触器 + 电机启动，进入 FSM RUNNING。

5
小脑 executor

追平层 + 站点吸附

监听 level_up / level_down 平层信号。
到达 5 楼 → 减速刹车 + 站点吸附（可能反冲修正几次）→ 状态切到 IDLE。

6
小脑 → 大脑 → 全栈

广播"动作完成"

Car.position = 5、Car.direction = IDLE、广播完成事件。
大脑用户交互模块：清 pending_calls → 决定是否要 OPEN_DOOR → 进入下一轮。

全程没有：await sleep() · while: poll · 跨层硬调用。
全部靠：事件广播总线 + ActionQueue + 传感器 listener 协同。
§6

Action 契约表brain ↔ cerebellum language

Action 是大脑算法层的输出语言，也是大脑与小脑之间的通信协议。 完整定义见 core/actions.py:15-27。

ActionKind 发出方 完成判据 小脑展开要点
INITIALIZE算法层（仅 state==UNKNOWN）触限位反向 + 平层计数完成全速跑方向 → 触 1 限位 → 反向逐层平层
MOVE_UP / MOVE_DOWN算法层平层匹配目标楼，direction=IDLE启电机+方向 → 过平层 → 减速刹车 → 反冲
OPEN_DOOR大脑用户交互door_open_done 信号拉开门继电器 → 等到位 → cron 兜底 8s
CLOSE_DOOR大脑用户交互（cron 闹钟响后）door_close_done 信号拉关门继电器 → 等到位 → cron 兜底 8s
SET_DISPLAY大脑IO 写入完成查 floor_display → 写 segment_* 位
EMERGENCY_STOP小脑（限位触发）全部 output 归零停电机 + 全刹 + 清所有输出
RESET_FAULT大脑（/clear 等）FaultFlags 全清清故障指示 + 恢复 ready
NOOP任意立即 done占位 / 心跳

语义约束

  • ▸ MOVE_UP / MOVE_DOWN 不碰门——算法层只负责把车开到目标楼，开关门是另一个独立子系统
  • ▸ OPEN_DOOR / CLOSE_DOOR 不碰电机——门控和运动在 executor 内部互斥（运动中拒绝开门）
  • ▸ INITIALIZE 期间锁所有——初始化过程中不响应其他 Action
  • ▸ EMERGENCY_STOP 最高优先级——任何层广播急停，小脑立即全输出归零
